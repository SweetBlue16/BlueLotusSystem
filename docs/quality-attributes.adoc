=== Escenarios de atributos de calidad
==== Disponibilidad
Según el estándar ISO/IEC 25010:2023, la disponibilidad es la capacidad de un producto para estar operativo y accesible cuando se requiere para su uso. Este atributo es crucial, ya que el sistema de reservaciones debe estar disponible en todo momento, ya que los huéspedes pueden realizar reservaciones o cancelaciones desde cualquier parte del mundo y en cualquier horario. Una caída del sistema impactaría directamente en la operación del hotel y la experiencia de usuario. Garantizar alta disponibilidad asegura la continuidad del servicio y confianza de los clientes.

==== Usabilidad
Según el estándar ISO/IEC 25010:2023, la usabilidad es el grado en el que un sistema, producto o servicio puede ser utilizado por usuarios específicos para lograr objetivos específicos con eficacia, eficiencia y satisfacción en un contexto de uso específico. En el contexto del sistema de reservaciones de un hotel, la usabilidad es fundamental debido a que el público objetivo incluye huéspedes con distintos niveles de experiencia tecnológica, recepcionistas que requieren rapidez en su trabajo y administradores que necesitan gestión clara de datos. Un sistema usable reduce la curva de aprendizaje, disminuye errores humanos y mejora la satisfacción tanto del cliente como del personal interno.

==== Rendimiento
De acuerdo con el estándar ISO/IEC/IEEE 24765:2017, el rendimiento se define como el grado en el que un sistema o componente cumple sus funciones designadas dentro de restricciones dadas, como velocidad, precisión o uso de memoria. En este contexto, el sistema debe responder de manera rápida y eficiente, especialmente durante la consulta de disponibilidad, generación de facturas o reportes. Un bajo rendimiento puede provocar que el huésped abandone la plataforma y busque alternativas en la competencia. Además, para el personal del hotel, la velocidad en check-in y check-out es crítica para evitar filas y retrasos en la atención.

==== Escalabilidad
Siguiendo el estándar ISO/IEC 25010:2023, la escalabilidad es definida como la capacidad de un producto para manejar cargas de trabajo crecientes o decrecientes o para adaptar su capacidad para manejar variabilidad. Para el sistema de reservaciones de un hotel, la plataforma debe poder crecer en función del aumento de hoteles, usuarios y transacciones sin degradar su desempeño. Esto es esencial si se expande a nivel regional o internacional, ya que debe adaptarse a mayor carga de consultas y operaciones sin necesidades de rediseños completos.

==== Seguridad
Basándose en el estándar ISO/IEC/IEEE 15288:2023, la seguridad se refiere a la protección contra subversión intencional o fallo forzado. Este atributo es crítico porque el sistema maneja datos sensibles como información de huéspedes, métodos de pago y reportes financieros. Una vulnerabilidad podría comprometer la privacidad de los clientes y la reputación del hotel. Implementar autenticación robusta, cifrado y control de accesos es clave para proteger la integridad de la información y evitar fraudes.

=== Restricciones y preocupaciones
==== Restricciones

[cols="^20, <30, <30, ^20", options="header"]
|===
| ID | Restricción | Descripción | Origen
| CON-01 | Uso obligatorio de base de datos relacional (RDBMS) | El sistema debe almacenar todas las transacciones y registros en un gestor relacional para garantizar consistencia, trazabilidad y control de concurrencia. | Técnico.
| CON-02 | Modelo de pago completo al reservar | El huésped debe cubrir el monto total de su estancia al confirmar la reserva, no se aceptan pagos parciales. | Negocio.
| CON-03 | Política de Overbooking | Se permite un máximo del 10% de sobreventa por tipo de habitación y fecha para optimizar la ocupación sin comprometer la experiencia del huésped. | Negocio.
| CON-04 | Cumplimiento legal de privacidad | El sistema debe cumplir con la Ley Federal de Protección de Datos Personales y estándares internacionales (ej. ISO/IEC 27001). | Legal.
| CON-05 | Alta disponibilidad | La arquitectura debe garantizar al menos 95% de disponibilidad, incluyendo redundancia y recuperación automática de servicios. | Técnico.
| CON-06 | Integración con pasarela de pagos externa | El sistema no procesará pagos directamente; se comunicará con un servicio externo certificado (Stripe, PayPal, etc.) mediante API segura. | Técnico.
| CON-07 | Arquitectura Cliente-Servidor | El sistema operará bajo un modelo cliente-servidor, donde el cliente (web/móvil) consume servicios a través de una API RESTful centralizada. | Técnico.
| CON-08 | Modelo en capas | El diseño se organizará en capas: Presentación, Lógica de Negocio, Acceso a Datos y API REST, para asegurar mantenibilidad y separación de responsabilidades. | Técnico.
| CON-09 | Comunicación segura | Todas las comunicaciones entre cliente y servidor deben realizarse mediante HTTPS con cifrado TLS 1.3. | Técnico.
| CON-10 | Registro obligatorio de transacciones | Todas las operaciones críticas (reservas, pagos, cancelaciones, auditorías) deben registrarse con sellado de tiempo para trazabilidad. | Legal.
| CON-11 | Multiplataforma web y móvil | El sistema debe funcionar en navegadores modernos y apps móviles (Android/iOS) que consuman la misma API REST. | Técnico.
| CON-12 | Desacoplamiento entre cliente y servidor | Ninguna operación de negocio o acceso a base de datos se ejecutará en el cliente; toda la lógica se gestiona del lado del servidor. | Técnico.
| CON-13 | Rendimiento mínimo garantizado | La API REST debe responder confirmaciones de reserva en ≤2.5 s (P95) bajo carga normal y ≤4.0 s (P99) en picos de tráfico. | Técnico.
| CON-14 | Política de recuperación ante desastres | Se debe contar con respaldos automáticos y capacidad de restaurar operación en ≤24 horas ante fallos graves. | Técnico.
| CON-15 | Despliegue modular y escalable | Cada capa (frontend, backend y base de datos) debe poder desplegarse y escalar de manera independiente en entornos cloud. | Técnico.
|===

==== Preocupaciones

[cols="^20, <20, ^60", options="header"]
|===
| ID | Preocupación | Descripción
| CRN-01 | Centralización operativa | El sistema debe permitir administrar hoteles, habitaciones y políticas de negocio desde un punto centralizado accesible vía web y móvil.
| CRN-02 | Experiencia fluida del usuario | La interfaz SPA debe ofrecer una navegación continua y responsiva, evitando recargas completas de página y mostrando mensajes contextuales inmediatos.
| CRN-03 | Seguridad de datos y autenticación | Protección integral de datos personales y financieros mediante autenticación multifactor (MFA), cifrado TLS, y control de accesos por roles.
| CRN-04 | Escalabilidad progresiva | La infraestructura debe soportar el crecimiento de usuarios, hoteles y transacciones mediante escalabilidad horizontal (nuevas instancias) y vertical (mayor capacidad de cómputo).
| CRN-05 | Mantenibilidad por capas | Cada capa debe poder modificarse, probarse y desplegarse sin afectar a las demás, facilitando actualizaciones y correcciones.
| CRN-06 | Eficiencia en la comunicación cliente-servidor | Las peticiones y respuestas API deben minimizar latencia y consumo de ancho de banda, especialmente en redes móviles.
| CRN-07 | Disponibilidad continua | El sistema debe mantener sus servicios operativos incluso durante actualizaciones, con mecanismos de failover y balanceo de carga.
| CRN-08 | Observabilidad y monitoreo | El sistema debe registrar métricas de uso, rendimiento y errores para análisis y diagnóstico, facilitando la detección temprana de fallos.
| CRN-09 | Integración con servicios externos | Debe ser posible conectar el sistema con APIs de terceros (pasarelas de pago, correo electrónico, agencias de viaje) sin afectar la operación principal.
| CRN-10 | Consistencia e integridad de datos | Las operaciones de reserva, pago y cancelación deben ser atómicas, garantizando que los datos reflejen el mismo estado en todos los componentes.
| CRN-11 | Recuperación ante fallos | El sistema debe restaurar servicios críticos de manera automática en ≤60 segundos, sin pérdida de información o sesiones activas.
| CRN-12 | Despliegue controlado | Los cambios en backend o base de datos deben poder revertirse (rollback) de forma segura y auditable.
| CRN-13 | Separación de responsabilidades | Definir claramente las funciones de cada capa: interfaz (cliente), negocio (servidor) y persistencia (base de datos).
| CRN-14 | Compatibilidad con dispositivos móviles | La interfaz debe adaptarse dinámicamente a distintos tamaños de pantalla y mantener el mismo rendimiento en dispositivos de gama media o baja.
| CRN-15 | Evolución tecnológica | La arquitectura debe permitir incorporar nuevos frameworks o servicios (como microservicios o GraphQL) sin afectar el núcleo del sistema.
|===
